# TDD Rule

> **이 규칙은 Rigid다. 상황에 맞게 변형하지 마라.** 예외가 필요하다고 판단되면 규칙을 어기지 말고, 왜 예외가 필요한지 사용자에게 먼저 확인하라.

## 0. 우선순위

이 파일은 **기본값(fallback)** 이다. 프로젝트 루트의 `CLAUDE.md` 또는 `AGENTS.md`에 TDD 관련 규칙이 명시되어 있다면 **그쪽이 우선**한다. 충돌 시 이 파일을 따르지 마라.

## 1. 적용 기준

### TDD를 반드시 적용해야 하는 대상

- **비즈니스 로직** (도메인 규칙, 계산, 상태 전이, 검증 로직)
- **API** (엔드포인트 핸들러, 요청/응답 처리, 에러 처리 경로)
- **유틸리티 함수** (순수 함수, 데이터 변환, 파서)
- **버그 수정** (수정 전에 버그를 재현하는 실패 테스트부터 작성)

### TDD가 불필요한 대상

- **타입 정의** (인터페이스, 타입 alias — 실행되는 동작이 없음)
- **설정 파일** (config, 빌드 스크립트, 환경변수 정의)
- **순수 UI** (마크업/스타일만 있고 로직이 없는 프레젠테이션 컴포넌트)
- **SQL** (스키마, 마이그레이션 — 별도의 마이그레이션 검증 절차를 따른다)

애매한 경우: 대상에 **분기(if/else)나 계산이 하나라도 있으면** TDD 적용 대상으로 간주한다.

## 2. RED-GREEN-REFACTOR 사이클

### RED

- **하나의 동작 = 하나의 테스트.** 여러 동작을 한 테스트에 묶지 마라.
- 테스트를 작성한 뒤 **반드시 실행해서 실패를 눈으로 확인**한다. 실행하지 않고 다음 단계로 넘어가는 것은 금지.
- 실패 이유는 오직 **"기능이 아직 구현되지 않음"** 이어야 한다. 문법 오류, import 오류, 오타로 인한 실패는 실패로 인정하지 않는다 — 고치고 다시 RED부터 확인하라.

### GREEN

- 테스트를 통과시키는 데 필요한 **최소한의 코드만** 작성한다.
- **YAGNI.** 지금 테스트가 요구하지 않는 기능, 매개변수, 분기를 미리 만들지 마라.
- 신규 테스트뿐 아니라 **기존 테스트 전체가 통과**하는지 확인한다.

### REFACTOR

- 중복 제거, 이름 개선, 헬퍼 추출만 수행한다.
- 매 변경 후 **테스트가 green을 유지**하는지 확인한다.
- 이 단계에서 **새로운 동작을 추가하지 마라.** 새 동작이 필요하면 다음 RED로 넘겨라.

### 반복

REFACTOR가 끝나면 다음 동작에 대한 **RED로 돌아간다.** 사이클을 건너뛰지 마라.

## 3. 삭제 강제 규칙

- 테스트보다 프로덕션 코드를 먼저 작성한 사실을 인지했다면, **그 코드를 삭제**하고 RED부터 다시 시작한다.
- "일단 참고용으로 남겨두기"도 금지한다. 삭제하지 않고 테스트만 나중에 끼워 맞추는 것은 TDD가 아니다.

## 4. 변명 차단표

| 변명 | 반론 |
|---|---|
| "너무 단순해서 테스트 불필요" | 단순한 코드일수록 테스트 작성 비용도 낮다. 단순함은 생략의 근거가 아니다. |
| "나중에 추가하겠다" | "나중"은 오지 않는다. 커버리지 없는 코드는 그대로 굳어져 리팩토링을 막는다. |
| "시간이 없다" | 테스트 없이 짠 코드는 디버깅 시간으로 그 차액을 반드시 되갚는다. 지금 쓰는 시간이 더 싸다. |
| "삭제하면 낭비" | 순서를 어기고 짠 코드는 검증되지 않은 코드다. 이미 손실은 발생했다 — 지금 삭제하는 것은 낭비가 아니라 손절이다. |
| "프로토타입이다" | 프로토타입은 살아남아 프로덕션이 되는 경우가 더 많다. 정말 버릴 코드라면 애초에 프로덕션 브랜치에 두지 마라. |
