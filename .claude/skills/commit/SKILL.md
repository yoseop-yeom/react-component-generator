# commit: 논리 단위 분할 + 한국어 커밋

현재 git 저장소의 변경사항을 사람이 리뷰하기 좋은 자연스러운 단위로 나누고, 단위마다 한국어
커밋 메시지를 붙여 순서대로 커밋한다. 커밋은 로컬 작업이라 되돌리기 쉽지만, 메시지와 분할
기준은 판단이 필요하므로 실제로 `git commit`을 실행하기 전에 계획을 사용자에게 보여주고
확인받는다.

## Step 0: 저장소 지침 확인

저장소 루트의 `AGENTS.md` 또는 `CLAUDE.md`를 찾아 커밋 컨벤션(타입 접두어, 스코프 표기,
언어, 본문 형식 등)을 명시하고 있는지 확인한다. 있으면 이 스킬의 기본 규칙보다 우선 적용한다.
모노레포처럼 하위 디렉토리에 별도 지침이 있으면 변경된 파일에 가장 가까운 지침을 따른다.

## Step 1: 변경사항 전체 파악

다음을 모두 확인해 전체 변경 범위를 파악한다.

- `git status`로 스테이징/미스테이징/untracked 파일 목록
- `git diff`와 `git diff --cached`로 실제 변경 내용
- `git log --oneline -20`으로 이 저장소의 기존 커밋 메시지 스타일(타입 접두어 사용 여부,
  언어, 길이 등)

변경이 전혀 없으면(스테이징·미스테이징·untracked 모두 없음) 그 사실을 안내하고 종료한다.

**민감 파일 경계**: `.env*`, `*.pem`, `*credentials*`, `*secret*`, `id_rsa` 류처럼 비밀값이
들어있을 수 있는 파일이 변경/추가 목록에 있으면 자동으로 포함하지 말고 사용자에게 알려
포함 여부를 직접 확인받는다.

## Step 2: 커밋 메시지 컨벤션 결정

- Step 0에서 저장소 자체 컨벤션을 찾았으면 그것을 그대로 따른다.
- 없고 기존 커밋 로그에서 일관된 스타일이 보이면(예: 항상 `fix:` 같은 타입 접두어를 쓴다)
  그 스타일을 이어간다.
- 아무 근거도 없으면 기본값으로 **Conventional Commits + 한국어 요약**을 쓴다:
  `<type>: <한국어 요약>`
  - `type` ∈ `feat`, `fix`, `refactor`, `style`, `docs`, `test`, `chore`, `perf`, `build`, `ci`
  - 요약은 명령형·현재형으로, 끝에 마침표를 붙이지 않는다 (예: `fix: 로그인 폼 빈 값 검증 누락 수정`)
  - 무엇을 바꿨는지는 diff와 파일명으로 이미 드러나므로, 요약은 **왜/무엇을 위해** 바꿨는지
    드러나게 쓴다. 필요하면 본문 줄을 추가해 비직관적인 이유(우회 조치, 숨은 제약, 트레이드오프)를
    설명한다. 자명한 내용은 본문에 반복하지 않는다.

## Step 3: 논리 단위로 분할

변경사항을 하나의 커밋에 몰아넣지 않고, 다음 기준으로 묶는다.

- **관심사 단위로 묶기**: 같은 기능/버그 수정/리팩터링/문서/설정/스타일 변경끼리 묶는다.
  서로 무관한 관심사는 분리한다.
- **의존 순서 고려**: 리팩터링 위에 기능을 얹는 식으로 순서 의존성이 있으면, 의존관계를
  거스르지 않는 순서로 커밋 순서를 정한다.
- **파일 내부 혼재 변경 분리**: 한 파일 안에 서로 무관한 변경(예: 기능 추가 + 포맷팅 정리)이
  섞여 있으면 파일 전체를 한 커밋에 넣지 말고 `git add -p`로 훅 단위 스테이징을 써서 분리한다.
- **기계적 변경은 따로**: 포맷터/린터가 만든 변경, import 정렬처럼 의미 없는 변경은 별도의
  `style`/`chore` 커밋으로 분리해 기능 변경 diff를 깨끗하게 유지한다.
- **과도한 세분화 지양**: 리뷰하기 좋은 자연스러운 단위가 기준이다. 서로 강하게 연관된 한두
  줄짜리 변경을 억지로 여러 커밋으로 쪼개지 않는다. 변경 전체가 원래 한 가지 목적이면 커밋
  하나로 둔다.
- 사용자가 이 스킬을 호출하며 분할 기준이나 범위를 함께 알려줬다면(예: 특정 파일만, 특정
  기능 단위로) 그 지시를 우선한다.

## Step 4: 계획 제시 및 확인

실제로 `git add`/`git commit`을 실행하기 전에, 아래 형식으로 커밋 계획을 보여준다.

```
커밋 계획 (총 N개)

1. <type>: <한국어 요약>
   파일: path/a.ts, path/b.ts
   (필요 시) 본문: <왜 이렇게 바꿨는지>

2. <type>: <한국어 요약>
   파일: path/c.ts
```

사용자가 명확히 "확인 없이 진행해", "바로 커밋해" 같이 자동 진행을 요청한 경우가 아니면,
계획을 제시한 뒤 진행 여부(전체 승인/일부 수정/취소)를 확인받는다. 사용자가 분할이나 메시지
수정을 요청하면 계획을 고쳐 다시 확인받는다.

## Step 5: 순차 커밋 실행

승인된 계획대로 커밋 단위마다 순서대로 실행한다.

1. 해당 단위에 속하는 파일만 `git add <구체적 경로>`로 스테이징한다. `git add -A`/`git add .`
   같은 전체 스테이징은 쓰지 않는다. 파일 내부 훅 분리가 필요한 단위는 `git add -p`로
   처리한다.
2. 커밋 메시지는 항상 heredoc으로 전달해 줄바꿈과 인코딩이 깨지지 않게 한다:
   ```bash
   git commit -m "$(cat <<'EOF'
   <type>: <한국어 요약>

   <본문(필요할 때만)>
   EOF
   )"
   ```
3. `--no-verify`, `--no-gpg-sign`, `--amend`는 사용자가 명시적으로 요청하지 않으면 쓰지
   않는다. pre-commit 훅이 실패하면 원인을 고치고 다시 스테이징해 **새 커밋**으로 만든다
   (직전 커밋을 amend하지 않는다).
4. 각 커밋 이후 `git status`로 다음 단위가 의도한 파일만 남아있는지 확인하고 다음 단위로
   넘어간다.

## Step 6: 완료 보고

모든 단위 커밋이 끝나면 `git log --oneline -N`(N=이번에 만든 커밋 수)으로 결과를 보여주고
간단히 요약한다. 사용자가 명시적으로 요청하지 않았다면 원격에 push하지 않는다.
